# Procession 3D 数字文明生成系统设计

> **文档身份**：本文件定义 Procession 如何将原始系统数据转化为一个可探索、有生命、有历史的数字文明。它是世界生成层的宪法级规范，所有 3D 场景、建筑、区域、生态、时间回溯的实现都必须回到这里验证：它是否让用户更相信自己正在观察一个真实存在的数字世界。
>
> **版本**：v1.0  
> **最后更新**：2026-07-18  
> **关联文档**：[`UI_DESIGN_SYSTEM.md`](UI_DESIGN_SYSTEM.md)、[`ARCHITECTURE.md`](ARCHITECTURE.md)、[`SPEC.md`](SPEC.md)

## 1. 设计宣言：电脑不是机器，是一颗数字星球

Procession 的世界生成系统必须回答一个问题：

**“如果我的电脑内部真的存在一个文明，它会是什么样子？”**

因此：

- **每一个系统实体都是生命体**，不是图标、不是方块、不是数据条。
- **每一次状态变化都是世界事件**，不是数字更新。
- **每一个用户看到的城市都独一无二**，因为每个人的电脑运行状态不同。
- **空间必须具有意义**：距离代表关系，高度代表强度，亮度代表活跃度，密度代表复杂度。
- **时间必须被保存**：城市有记忆，可以回溯，可以观察历史。

---

## 2. 核心设计原则

### 2.1 世界优先于数据

用户永远不会先看到“CPU 82%”。他先看到的是城市能源网络增强、建筑长高、空气热浪。数字只是对世界现象的确认。

### 2.2 空间即语义

| 空间属性 | 系统含义 |
|----------|----------|
| 距离中心远近 | 与系统核心的关系（内核近，边缘应用远） |
| 建筑高度 | 资源占用与活跃程度 |
| 建筑面积 / 体积 | 复杂度与影响力 |
| 亮度 | 当前能量活动 |
| 颜色温度 | 进程类型与身份 |
| 连接密度 | 网络与进程间通信强度 |
| 区域纹理 | 应用程序类别与文明特征 |

### 2.3 生长，而非出现

建筑、区域、连接都不是瞬间生成的。它们必须经历：

1. 数据粒子聚合
2. 能源连接建立
3. 结构形成与稳定
4. 持续呼吸与活动

### 2.4 死亡，而非删除

进程关闭时：

1. 能源供应断开
2. 内部光流熄灭
3. 数据连接消散
4. 结构逐渐解构
5. 残余能量回归城市生态

### 2.5 历史是世界的记忆

城市不是每次打开重新生成的动画。它保存状态快照，用户可以时间旅行，观察文明的兴衰。

## 3. 多层空间结构

Procession 的世界由四个连续层级构成，用户可以在它们之间平滑缩放， never 切换页面。

```
Universe Layer（宇宙层）
    ↓ 缩放进入
Civilization Layer（文明层 / 城市层）
    ↓ 缩放进入
District Layer（区域层）
    ↓ 缩放进入
Architecture Layer（建筑层 / 进程内部）
```

### 3.1 Universe Layer — 数字星球

**用户第一眼看到的整体视角。**

这不是天空盒，而是一片由系统数据构成的数字空间：

- **数据星尘**：漂浮的微小粒子，代表系统中的空闲资源与背景活动。
- **神经网络结构**：缓慢变化的连线网络，象征系统内部的数据通道。
- **能量轨道**：环绕城市的光带，由网络流量驱动。
- **数据流星**：偶尔划过空间的光点，代表突发的网络包或事件。
- **系统核心光源**：位于世界中心的稳定光源，象征 CPU / 内核。

**规模**：约 2000–5000 单位半径。

**交互**：用户可以远距离观察整个文明的能量状态，看到哪些区域在活跃、哪些在沉睡。

### 3.2 Civilization Layer — 数字城市

**根据真实系统状态动态生成的城市。**

城市不是固定地图。其布局由以下数据驱动：

| 数据源 | 城市表现 |
|--------|----------|
| CPU | 能源区域密度、工业光芒强度、空气热浪 |
| Memory | 建筑规模、空间扩张程度 |
| Process 数量 | 人口密度、区域数量 |
| Network | 道路与数据交通网络密度 |
| Storage | 仓储区域、资源结构、沉积层 |
| GPU | 特殊生态区、极光与能量现象 |
| Uptime | 城市成熟度、基础设施完整度 |

**规模**：约 200–800 单位半径。

**布局原则**：

- 系统核心服务位于城市中央，形成稳定几何结构。
- 用户应用程序按家族聚集，形成不同文明区域。
- 高资源消耗程序形成更大规模的地标建筑。
- 长期运行程序的建筑更成熟、结构更完整。
- 频繁启停的程序形成城市边缘的新兴、不稳定区域。

### 3.3 District Layer — 文明区域

**不同类型系统实体形成的独特文明区域。**

每个区域都有自己的建筑语言、运动节奏、光效特征和氛围。

#### Kernel District（内核区）

- **代表**：操作系统核心进程。
- **特征**：
  - 城市正中央的巨大中央核心塔。
  - 稳定、低频、几何精确的结构。
  - 持续能源循环，像心跳一样脉动。
  - 冷白金属与深灰材质。
- **建筑形态**：多环同心塔、放射状能量通道、厚重的基座。

#### Application District（应用区）

不同应用拥有不同数字身份：

| 应用类型 | 文明身份 | 建筑语言 |
|----------|----------|----------|
| 浏览器（Chrome / Edge / Safari） | 信息商业都市 | 玻璃幕墙、密集信息节点、数据高速公路、不断扩张的街区 |
| 代码编辑器（VS Code / JetBrains） | 工业研发中心 | 编译工厂、数据管线、工程设施、冷灰钛金属 |
| 游戏 | 模拟生态区域 | 动态环境、高变化性、大规模能源消耗、有机与几何混合 |
| 媒体 / 设计工具 | 创意工坊区 | 流动形态、暖色能量、创作残影 |
| 系统服务 | 基础设施管网 | 规则重复结构、低亮度、稳定脉动 |
| 后台工具 / CLI | 地下工程区 | 紧凑、功能主义、隐藏于城市下层 |

### 3.4 Architecture Layer — 建筑内部

**单个进程不是一栋建筑编号，而是一个数字生命体。**

当用户进入建筑，他将看到：

- **内部能源核心**：CPU 活动产生的光球或反应堆。
- **资源管线**：内存分配形成的流动管道。
- **线程人群**：活跃线程像工作人员一样移动。
- **数据流**：网络 IO 像血液一样在建筑内循环。
- **子进程腔室**：子进程像附属设施或器官一样分布。

## 4. 数据到文明的映射

### 4.1 建筑属性映射

每个进程建筑由真实数据驱动，禁止静态模型。

| 数据输入 | 建筑属性 | 映射说明 |
|----------|----------|----------|
| Memory 占用 | 建筑高度 | `height = f(mem_mb)`，高内存 = 高塔 |
| Process 复杂度（线程数 + 句柄数） | 建筑面积 / 体积 | 复杂程序更庞大 |
| CPU 活跃度 | 建筑亮度与能量脉动 | 高 CPU = 强光、快呼吸 |
| Thread 数量 | 内部人口 / 活动粒子 | 线程像工作人员 |
| IO + Network 活动 | 能源流速度与密度 | 高 IO = 管道高速流动 |
| 运行时间 | 建筑年龄与成熟度 | 老程序结构更完整、颜色更沉稳 |
| 历史崩溃次数 | 建筑稳定性 / 损伤痕迹 | 崩溃过的建筑有裂纹或暗斑 |
| 进程树深度 | 距市中心距离 | 核心服务近，深层子进程远 |
| 父进程关系 | 区域聚集与连接 | 同家族进程形成街区 |

### 4.2 复杂度函数

```typescript
function processComplexity(p: ProcessInfo): number {
  const threadWeight = p.threads ?? 1;
  const handleWeight = (p.handle_count ?? 1) / 100;
  const childWeight = (p.child_count ?? 0) * 0.5;
  return Math.log1p(threadWeight + handleWeight + childWeight);
}
```

> 使用 `log1p` 避免极少数极端进程（如浏览器主进程）破坏城市尺度。

### 4.3 年龄与成熟度

```typescript
function buildingMaturity(uptimeSeconds: number): number {
  // 0 = 刚刚启动，1 = 完全成熟
  return Math.min(1, uptimeSeconds / 3600); // 1 小时达到完全成熟
}
```

- 年轻建筑：结构更简单、材质更亮、呼吸更快（像新生儿）。
- 成熟建筑：结构完整、材质沉稳、节奏稳定。

## 5. 程序化城市生成

### 5.1 生成流程

```
SystemSnapshot
    ↓
1. 数据解析与分类
    ↓
2. 区域分配（内核区 / 应用区 / 特殊区）
    ↓
3. 区域内布局算法
    ↓
4. 建筑实例化与属性赋值
    ↓
5. 连接网络生成（父子关系、网络连接）
    ↓
6. 环境与天气系统更新
    ↓
Rendered World
```

### 5.2 区域布局算法

#### 内核区布局

- 采用**极坐标放射布局**。
- 中心为 Kernel Core Tower。
- 核心服务围绕中心按优先级环形分布。
- 间距规则、几何稳定。

```typescript
function kernelLayout(processes: ProcessInfo[]) {
  return processes.map((p, i) => {
    const angle = (i / processes.length) * Math.PI * 2;
    const radius = 20 + p.priority * 5; // 高优先级更靠近核心
    return {
      x: Math.cos(angle) * radius,
      z: Math.sin(angle) * radius,
    };
  });
}
```

#### 应用区布局

- 采用**力导向聚类 + 螺旋采样**。
- 同家族进程聚集。
- 高资源进程占据更中心位置。
- 新进程从区域边缘“生长”进入。

```typescript
function applicationLayout(district: ProcessInfo[]) {
  // 1. 按应用家族分组
  const groups = groupByAppFamily(district);
  // 2. 为每个家族分配一个扇形区域
  // 3. 在家族内部使用螺旋 + 抖动分布
  // 4. 高资源进程获得更好的“地段”（更靠近区域中心）
}
```

### 5.3 位置稳定性

为避免建筑每帧跳动：

- 每个进程分配一个**稳定的种子位置**，基于 PID + 应用名哈希。
- 实际位置向种子位置弹性插值，而不是瞬间跳转。
- 进程关闭后，位置不会立即被新进程占据，而是保留为“遗址”一段时间。

```typescript
// 伪代码
const targetPosition = computeTargetPosition(process);
const currentPosition = lerp(currentPosition, targetPosition, 0.1); // 每秒向目标移动 10%
```

## 6. 建筑生成系统

### 6.1 建筑不是模型，是生命体

禁止加载静态 .gltf / .obj 模型作为建筑。建筑必须由代码程序化生成，并随数据变化。

### 6.2 建筑结构组成

一个建筑由以下元素构成：

| 元素 | 含义 | 数据驱动 |
|------|------|----------|
| 基座 | 进程稳定性与历史 | 年龄、崩溃次数 |
| 主体 | 资源规模 | 内存、线程数 |
| 顶部 | 当前活跃度 | CPU |
| 外壳 | 进程类型身份 | 应用分类 |
| 内部光核 | 能源活动 | CPU + IO |
| 连接端口 | 网络与父子连接 | 连接数 |
| 呼吸光环 | 生命周期节奏 | 运行时间、状态 |

### 6.3 建筑生长动画

新进程诞生时，建筑按以下阶段生长：

1. **0–300ms**：数据粒子从地面向目标位置聚合。
2. **300–900ms**：能源连接从周围建筑/核心延伸到新建筑。
3. **900–1500ms**：基座形成，主体向上生长。
4. **1500–2000ms**：外壳稳定，内部光核点亮，呼吸开始。

> 生长时长可根据系统负载动态调整：高负载时城市整体节奏加快。

### 6.4 建筑消亡动画

进程关闭时：

1. **0–300ms**：内部光核变暗，能源流减速。
2. **300–800ms**：连接端口关闭，数据流消散。
3. **800–1500ms**：外壳失去光泽，结构开始解构。
4. **1500–2500ms**：建筑坍塌为粒子，回归城市地面。

> 长期运行的老建筑消亡时应更缓慢、更有仪式感。

### 6.5 建筑状态语言

| 系统状态 | 建筑表现 |
|----------|----------|
| Running | 稳定呼吸、内部光流正常 |
| Sleeping | 呼吸极缓、亮度降低 |
| Stopped | 光核熄灭、外壳灰暗 |
| Zombie | 外壳破损、无规律抽搐、残影 |
| High CPU | 亮度增强、呼吸急促、顶部能量喷发 |
| High Memory | 体积膨胀、结构紧绷 |
| Network Burst | 连接端口高速闪烁、数据流密集 |

## 7. 生命周期系统

### 7.1 进程生命周期阶段

```
Conception（孕育）
    ↓ 数据首次出现
Birth（诞生）
    ↓ 建筑生长完成
Operation（运行）
    ↓ 持续活动、变化、成长
Aging（老化）
    ↓ 长期运行带来的稳定与疲惫
Death（死亡）
    ↓ 进程结束
Dissolution（消解）
    ↓ 能量回归生态
Memory（记忆）
    ↓ 历史快照中保留
```

### 7.2 生命周期可视化规则

- **诞生**：新建筑从区域边缘或父建筑附近生长出来。
- **运行**：建筑持续呼吸，内部有光流和人流。
- **老化**：颜色略微变暗，结构出现稳定感，偶尔有“维护脉冲”。
- **死亡**：完整消亡动画，不是瞬间删除。
- **消解**：建筑位置变为微弱的地面光斑，新进程可能在此“重生”。

### 7.3 关系网络

- **父子关系**：父建筑与子建筑之间有能源管线连接。
- **同家族关系**：同应用家族的进程之间有天线或数据通道连接。
- **网络连接**：跨进程的网络通信形成光缆。
- **竞争关系**：争夺相同资源的进程之间可能出现能量对抗（微妙的红光闪烁）。

## 8. 摄像机系统

### 8.1 设计目标

摄像机不是游戏自由镜头，而是**未来实验室的观察设备**：

- 平滑
- 缓慢
- 有空间重量感
- 从不硬切
- 每一次移动都像精密仪器在调整焦距

### 8.2 摄像层级

| 层级 | 距离 | 观察对象 |
|------|------|----------|
| Universe | 1000–3000 | 整个数字星球 |
| Civilization | 300–1000 | 城市整体布局 |
| District | 80–300 | 单个区域 |
| Architecture | 10–50 | 单个建筑内部 |
| Core | 0–10 | 系统核心底层结构 |

### 8.3 运动模式

#### Orbit（轨道观察）

- 默认模式。
- 摄像机围绕目标点缓慢公转。
- 用户拖拽可调整轨道角度。
- 释放后轨道运动有惯性衰减。

#### Drift（漂移）

- 无用户输入时，摄像机有极缓慢的自动漂移。
- 漂移速度与系统 CPU 成正比：高负载时城市“更不安分”。

#### Descent（下沉）

- 从 Universe 进入 Civilization 时的运动。
- 不是缩放，而是物理上的空间下沉。
- 伴随景深变化、粒子密度增加、细节逐渐显现。

#### Focus（聚焦）

- 用户点击建筑后，摄像机飞向该建筑。
- 移动时长 500–900ms，使用 Expo ease-out。
- 接近时世界其他部分轻微模糊，建筑外壳半透明化。

### 8.4 约束与边界

- 最小距离：防止穿透建筑内部造成迷失。
- 最大距离：防止用户飞出宇宙层。
- 过渡速度：由当前层级决定，越接近微观越慢。
- 禁用瞬间传送：所有层级切换必须是连续运动。

## 9. 时间系统与历史

### 9.1 世界必须有记忆

Procession 不是实时监控面板，而是一个有历史的城市。

系统需要保存：

- 城市布局变化
- 软件生命周期（诞生、消亡）
- 性能峰值与异常事件
- 用户交互历史（用户缩放、聚焦过哪里）

### 9.2 快照存储

```typescript
interface WorldSnapshot {
  timestamp: number;
  processes: ProcessSnapshot[];
  cpu: number;
  memory: number;
  network: NetworkSnapshot;
  events: WorldEvent[];
}
```

**存储策略**：

- 最近 1 小时：每秒 1 帧。
- 最近 24 小时：每 10 秒 1 帧。
- 最近 30 天：每小时 1 帧关键状态。
- 更久：每天 1 帧。

### 9.3 时间旅行交互

1. 用户选择 **History Lens**。
2. 底部出现一条时间轴，画面色调微微变冷 / 变旧。
3. 拖动时间轴：城市根据快照重构。
4. 释放时间轴：画面可选择恢复当前，或继续停留在历史状态。

### 9.4 历史事件标记

时间轴上标注关键事件：

- 新应用首次出现
- 性能峰值
- 进程崩溃
- 系统启动 / 唤醒
- 用户手动标记

### 9.5 可逆播放

- 支持正放、倒放、暂停。
- 建筑生长动画可逆播放：从成熟状态“倒带”回诞生状态。
- 消亡动画可逆播放：从消散状态“倒带”重建建筑。

## 10. 性能与扩展

### 10.1 渲染预算

| 指标 | 目标 |
|------|------|
| 目标帧率 | 60fps @ 1080p，30fps @ 4K |
| 建筑数量 | 默认 500，最高 2000 |
| Draw Calls | 建筑 < 5（InstancedMesh） |
| 粒子数量 | 5000–20000 GPU 粒子 |
| 后处理开销 | < 3ms / 帧 |

### 10.2 优化策略

- **GPU Instancing**：所有建筑使用 InstancedMesh 一次性渲染。
- **LOD 系统**：远距离建筑降级为光点或简化几何。
- **Frustum Culling**：宇宙层只渲染可见区域。
- **Compute Shader**：粒子系统与数据流使用 WebGPU Compute Shader。
- **纹理图集**：建筑材质使用统一图集，减少绑定切换。
- **差分更新**：只更新变化的数据，而不是每帧重建整个场景。

### 10.3 数据层到视觉层的严格分离

```
Data Layer（SystemSnapshot）
    ↓ 解析、过滤、聚合
Simulation Layer（WorldState）
    ↓ 插值、预测、生命周期管理
World Generation Layer（Scene Graph）
    ↓ 程序化生成建筑、区域、连接
Rendering Layer（R3F / Three.js / WebGPU）
    ↓ 相机、后处理、性能优化
Interaction Layer（用户输入、Lens Console、Entry Probe）
```
