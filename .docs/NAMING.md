# Procession 命名决策记录

> 命名是最重要的设计决策之一。好的名字让项目在千万 GitHub 仓库中被记住。
> 这份文档记录了我们如何从零开始命名的过程和思考。

## 约束条件

在开始之前，我们确定了命名的约束：

1. **短** — 2 个音节以内，容易发音，容易记住
2. **唯一** — GitHub 上没有成大器的同名项目
3. **含义自解释** — 开发者看到名字能隐约猜到项目做什么
4. **有品牌感** — 像 React、Vue、Node 一样能独立于字面被记住
5. **与项目调性匹配** — 桌面艺术品 x 系统监控，不是枯燥工具

## GitHub 100K+ Stars 命名规律分析

我们研究了 GitHub 上多个 stars 超过 10 万的代表性项目，归纳命名共性：

| 命名类型 | 代表项目 | 规律 |
|---------|---------|------|
| **简短造词** | react, vue, go, rust, node, linux, flutter, deno, electron | 1-2 音节，极易发音，品牌感极强 |
| **合成词** | tensorflow, kubernetes, freeCodeCamp, ohmyzsh, vscode | 两个词根融合，读起来流畅，含义自解释 |
| **核心词 + .js** | next.js, three.js, d3.js | 技术标识明显，已成为独立品牌 |
| **短语式** | create-react-app, awesome, You-Dont-Know-JS | 脚手架/清单/教程类，名字即用途 |

**规律核心：** "优秀的开源项目，名字都短、好读、有唯一性。要么是一个有双关或重定义的单词，要么是两个耳熟能详的词根巧妙拼接。" 极少直接使用"功能+对象"这类硬组合，因为那像是软件说明书，很难成为独立品牌。`tensorflow` 就比 `machinelearning-framework` 好一万倍。

## 候选方案分析

基于以上规律，我们产生了 4 个候选：

### SysCity

- 符合"组合词"模式，直白好懂
- 问题：City 被大量项目使用，独特性偏弱
- 缺乏双关或情绪张力，更像一个功能模块名

### SysPunk

- 造词，风格强烈
- 问题：Punk 已是过饱和的命名后缀，容易淹没在同类中

### Procscape

- 符合"两个词根融合"模式，类似 landscape 的构造
- Process + Scape（景象）= 进程景象
- 含义精准但直白，少了点诗性和记忆点，更像学术工具命名

### Procession ✅ 最终选择

- **单个单词**，8 字母，2 音节，极短极好记
- **双关完美**：
  - Process（进程）— 技术内核
  - Procession（巡游/行列）— 视觉叙事
- **独特性极高**：GitHub 上目前没有成名的同名项目
- **品牌感强**：像 React、Vue 一样，能脱离字面被记住和传播
- **调性匹配**：把冷冰冰的系统监控升格为动态图谱艺术，名字本身就是叙事的一部分
- **Slogan 自然**："进程列队，系统成诗"——宣传片画面感直接出来

## 结论

Procession 同时满足所有约束：

| 约束 | 结果 |
|------|------|
| 极简 | 一个单词，8 字母，2 音节 |
| 唯一性 | GitHub 无同名成名项目 |
| 含义双关 | Process × Procession |
| 品牌感 | 像 React、Vue 一样可独立记忆 |
| 调性匹配 | 名字本身就是叙事 |
