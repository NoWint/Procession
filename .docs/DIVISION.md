## 三、核心原则

> **后端只做事，不做视觉** — 严梓峻 只负责数据采集和推送，不碰任何渲染逻辑。
>
> **前端只消费，不调 Rust** — 夏天 只通过 IPC 事件消费数据，不直接调用 Rust 代码。

数据格式变更需要通知对方，双方同步更新 `types.rs` 和 `types.ts`。

## 四、IPC 事件清单

| 事件名 | 方向 | 频率 | 发送方 | 接收方 |
|--------|------|------|--------|--------|
| `system-snapshot` | 后端 → 前端 | 1Hz | 严梓峻 | 夏天 |
| 设置变更 | 前端 → 后端 | 按需 | 夏天（UI）→ 严梓峻（处理） | 双方 |
| 进程操作（kill） | 前端 → 后端 | 按需 | 夏天（UI）→ 严梓峻（实现） | 严梓峻 |

## 五、协作规范

### 代码审查

- 后端 PR → **严梓峻开，夏天审**（主要看接口契约是否清晰）
- 前端 PR → **夏天开，严梓峻审**（主要看 IPC 数据使用是否正确）
- 跨层变更（如 IPC 数据格式改动）**必须双方同时 review**

### 沟通机制

| 场景 | 方式 |
|------|------|
| 日常沟通 | 微信 / 飞书即时消息 |
| 设计决策 | 在对应 Issue/PR 下讨论，记录结论 |
| 接口变更 | 提前通知对方，同步更新类型文件 |

### 分支策略

- `main` — 稳定分支，保护不可直接推送
- `feat/<name>` — 功能分支，各自开发
- 适时合并，避免长期分支偏离

## 七、待决策项

以下事项需要双方协商确定：

1. **布局算法在哪侧计算？** 前端实时算（数据量小、灵活）vs 后端预处理（减少前端计算量）
2. **Phase 3 光缆映射** 对端 IP → 城市坐标的转换由后端预处理还是前端自行查找？
3. **配置持久化** Tauri 的 store plugin 还是自建 JSON 文件？
4. **CI/CD** 初期是否需要 GitHub Actions 自动化构建？
